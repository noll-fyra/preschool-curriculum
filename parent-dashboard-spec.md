# Nurture — Parent Dashboard: Design Specification
*Version 1.0 | Scope: MVP parent-facing mobile app*

---

## Purpose of this document

This specification describes the parent-facing dashboard in full — the design philosophy, information architecture, screen-by-screen behaviour, component details, and the reasoning behind every significant decision. It is written for an AI coding agent or developer implementing the feature from scratch.

Read this document in full before writing any code. The reasoning sections are not decorative — they encode constraints that will break the experience if ignored.

---

## The emotional design problem

The parent dashboard is not a data display. It is an anxiety-management tool that happens to contain data.

Parents of preschool children (ages 3–6) experience a specific form of anxiety: their child has entered an institution for the first time, is being shaped by people the parent doesn't fully know, and the parent has almost no visibility into what's happening. Every piece of information the dashboard surfaces will be received through this emotional lens.

The design must answer three questions a parent brings to every session:

1. **Is my child okay?** (reassurance — the 30-second check)
2. **What is my child actually working on?** (curiosity — the 3–5 minute exploration)
3. **Is there something I should be worried about or doing?** (concern — the prompted action)

The interface must serve all three without designing primarily for any one of them. A parent who only ever needs the 30-second check should never be forced to dig. A parent who wants to understand everything should be able to go as deep as the data allows.

---

## The three rules that govern all design decisions

**Rule 1: Progress is always relative to the child, never to peers.**
Never show class averages, percentile rankings, or any data point that implies comparison to other children. There are no leaderboards, no "ahead of class" or "behind class" labels. Every number shown refers only to what this child has done and where they are on their own journey. Violating this rule will trigger exactly the anxiety the product is meant to reduce.

**Rule 2: Data always comes with context.**
A raw score, level name, or milestone status shown without explanation will be misread. "Beginning" with no context sounds like failure. "3 of 5 sessions" with no context creates dread. Every piece of data must be accompanied by plain-language meaning — what it is, why it matters, what to expect next. The information architecture enforces this: raw data and its explanation are always co-located.

**Rule 3: The emotional headline comes before the data.**
Every screen entry point leads with a human-language summary before any structured data. The parent reads a sentence first, then sees numbers. This order is non-negotiable — reversing it (numbers first, explanation below) causes parents to react to the number before they have context to interpret it correctly.

## Navigation structure

The app has three top-level tabs:

| Tab | Purpose |
|---|---|
| Home | The dashboard — status overview, activity feed, entry point to all drill-downs |
| Activities | The child's current activity queue — for handing the device to the child |
| Messages | Async thread with the class teacher |

Navigation within the app is a simple push stack. Every drill-down screen has a back arrow returning to its parent screen. There are no modals, no bottom sheets, no overlapping layers. The hierarchy is:

```
Home
  ├── Learning area detail (LL / NUM / SED)
  │     └── Milestone detail
  ├── Activity result detail
  ├── Milestone achieved detail
  ├── Teacher note / message thread
  └── P1 readiness detail

Activities (tab)

Messages (tab)
  └── Full message thread with teacher
```

## Screen 1: Home

### Purpose
The home screen answers "Is my child okay?" in under 5 seconds, and then provides entry points for parents who want more.

### Layout (top to bottom)

**1. Status bar**
Standard mobile status bar. Time left, signal right. No customisation needed.

**2. Greeting + child name**
```
Good morning          ← muted secondary text, 13px
Aisha's dashboard     ← primary, 22px medium weight
```
The greeting is time-of-day aware: Good morning / Good afternoon / Good evening. The child name uses the possessive form pulled from the child profile. If a parent has multiple children, a child switcher appears here (post-MVP feature — for MVP, single child only).

**3. Hero card — the emotional headline**

This is the most important element on the screen. It is a coloured card (green background, green border) containing:
- A small label: "This week" in uppercase, 11px, spaced
- A 2–3 sentence plain-language summary of how the child is doing

The summary is generated by a simple rule engine (not AI in MVP) based on:
- Number of activities completed this week vs last week
- Whether any milestones were achieved
- Whether there are any areas with no recent activity (triggers a softer message)

**Tone rules for the summary text:**
- Always warm and specific — never generic ("great job!")
- Never alarming — even a quiet week is framed constructively
- Always ends with something forward-looking or actionable
- Maximum 2 sentences

Example outputs:
- Active week: "Aisha had a great week — she completed 5 activities and is close to her next reading milestone. Keep it up!"
- Quiet week: "It's been a quieter week for Aisha — no activity completions logged at home. A quick session this weekend could help keep her momentum going."
- Milestone week: "Aisha reached a big milestone this week — she now knows all 26 letters. She's ready to start working on letter sounds next."

The card is always green. It should never be red or amber — a negative-feeling hero card on entry will cause parents to avoid opening the app.

**4. Section label: "Learning areas"**

Plain uppercase label, 11px, tertiary text colour. No interactive element.

**5. Three area cards**

One card per learning area: Language & Literacy, Numeracy, Social & Emotional Development.

Each card contains:
- Area name (14px, medium)
- Level badge (pill): Beginning / Developing / Secure — colour-coded amber/amber/green respectively (see colours section)
- Progress bar: a thin horizontal track (6px tall) with a filled portion indicating rough position within the current level. This is NOT a precise percentage — it is a gestural indicator showing relative position. Fill values:
  - Beginning: 0–33% of bar filled
  - Developing: 34–66% of bar filled
  - Secure: 67–100% of bar filled
  - Within each level, the fill moves as milestones are achieved (Beginning with 0/3 milestones = near-empty, Beginning with 3/3 = near the Developing threshold)
- Current focus: "Working on: [plain-language description of current in-progress milestone]"
- Recency: "Last activity: today / yesterday / [day name] / [X days ago]"

The entire card is tappable and navigates to the learning area detail screen. Tap target is the full card. Visual affordance: subtle hover/press state (background shifts to secondary).

**Critical: the progress bar must never show a percentage number.** No "55%" label. The bar is gestural — it communicates direction and rough position, not precision. Precision triggers comparison anxiety.

**6. Section label: "This week"**

**7. Activity feed — 3 most recent items**

A vertical list of feed items, each showing:
- An icon (small coloured square with an SVG icon inside): blue for activity completions, green for milestone achievements, amber for teacher notes
- Title: what happened, in plain language
- Subtitle: additional context (score, which area, what it means)
- Timestamp + location ("Today, 3:45 pm · at home" or "at school")

All feed items are tappable and navigate to a detail screen.

Feed item types and their display logic:

| Type | Title format | Subtitle format |
|---|---|---|
| Activity completed | "Completed: [activity name in plain language]" | "Got [X] out of 3 right — [qualitative gloss]" |
| Milestone achieved | "Milestone reached!" | "[Child name] now [milestone statement in plain language]" |
| Teacher note | "Note from [teacher first name]" | First ~80 characters of the note, truncated with ellipsis |
| Behaviour observation | "Ms [name] noticed something" | "[Brief description of observed behaviour]" (if school opts to surface these) |

Show maximum 3 items on the home screen. A "See all" link below opens a full chronological feed (post-MVP).

**8. P1 readiness banner**

A tappable row at the bottom of the scrollable content area (above the fixed tab bar). Contains:
- Left: small label "P1 readiness" + value "X of Y milestones achieved"
- Right: "See details →" in info text colour

This is not part of the tab bar — it sits in the content scroll. It is persistent on the home screen because P1 readiness is the primary anxiety driver for Singapore parents and should always be one tap away.

**9. Fixed tab bar**

Three tabs pinned to the bottom: Home (house icon), Activities (checklist icon), Messages (speech bubble icon). Active tab uses info background on the icon. Inactive tabs use tertiary text colour. No badge counts in MVP (notification dots are post-MVP).

## Screen 2: Learning area detail

Reached by tapping any area card on the home screen.

### Purpose
Answers "Where exactly is my child on this journey?" Shows the full milestone sequence — achieved, in progress, and upcoming — for one learning area.

### Layout

**Top bar**
Back arrow (← ) + area name as title (17px medium). No additional controls.

**Area summary paragraph**
1–2 sentences explaining what this area is and where the child currently stands. Written in plain language. Example for Language & Literacy at Developing level:
> "Aisha is building the skills she needs to read independently. She's currently at the Developing level — working through the middle set of milestones."

This text is dynamically generated from a template with the child's name and current level slotted in. Keep the template warm and specific. Avoid clinical language.

**Full-width progress bar**
Larger version of the home screen bar (8px tall). Below it, three equidistant labels: "Beginning", "Developing", "Secure". This contextualises the bar as a journey from start to finish, not a score out of 100.

**Milestone list, grouped by level**

Three sections: Beginning, Developing, Secure.

Each section has a section header row containing:
- Level pill (coloured badge)
- Achievement summary: "✓ All 3 achieved" or "2 of 4 achieved" or "Not yet started"

Within each section, milestone rows:

Each milestone row contains:
- Icon circle (28px): green checkmark for achieved, amber dot for in-progress, grey lock for not-yet-started
- Milestone text: the plain-language statement
- Status subtext: "Achieved" / "In progress" / "Unlocks next"

**The currently active milestone is visually highlighted** — its row has a secondary background fill, its title is slightly heavier, and it shows the mastery progress ("3 of 5 passing sessions" or "3 of 5 observations"). This row also has a right arrow indicating it is tappable for more detail.

**All achieved milestones are tappable** and navigate to a simplified milestone detail screen (showing the explanation and when it was achieved). This lets curious parents understand what their child has already mastered.

**Locked milestones show their label in muted text** and are not tappable in MVP. They are visible so the parent can see the path ahead. This is intentional — seeing what's coming next reduces anxiety about the unknown. Do not hide locked milestones.

**The Secure level section is visually muted** (reduced opacity: 0.5) when the child has not yet reached Secure. This signals "future" without hiding the information.

**Special handling for Social & Emotional Development:**
This area has no activities. The area summary paragraph includes an explicit explanation:
> "Social and emotional development is assessed by Ms Priya through classroom observation — not activities. These milestones show what Aisha demonstrates consistently in real interactions with friends and teachers."

This is important. Parents must understand why there are no activity scores for this area, or they will wonder if something is broken. Include a closing note about how they can reinforce these behaviours at home, in plain language without jargon.

## Screen 3: Milestone detail (in-progress)

Reached by tapping the highlighted in-progress milestone row.

This is the most information-dense screen in the app and the most important one to get right. It answers the question parents most want answered: "What is my child actually working on, why does it matter, and what can I do?"

### Layout

**Top bar:** Back arrow + "Milestone detail"

**Header block:**
- Area + level label: "Language & Literacy · Developing" — small, muted, uppercase
- Milestone title: the full plain-language milestone statement, 18px medium, generous line-height
- Mastery progress bar + label: "3 of 5 sessions" — amber fill on the bar, amber text for the label

**"Why this matters" card**
Background: light blue (#E6F1FB), border: blue (#85B7EB)
Label: "WHY THIS MATTERS" in blue uppercase
Body: 3–5 sentences written for a non-specialist parent. Must explain:
1. What the skill actually is (defined plainly)
2. Why it matters for reading/maths/development (the educational significance)
3. What it leads to next (connects to the journey)

This text is authored content, not dynamically generated in MVP. One "why this matters" block per milestone, written by a curriculum specialist and stored as content in the database alongside the milestone record.

**"Try at home" card**
Background: light amber (#FAEEDA), border: amber (#EF9F27)
Label: "TRY AT HOME" in amber uppercase
Body: One specific, actionable suggestion for reinforcing this skill at home.

Rules for writing these suggestions:
- Must be completable in under 5 minutes
- Must require no materials (or only things found at home)
- Must be framed as a moment, not a task ("at the supermarket" / "while you're cooking" / "at bedtime")
- Must feel like good parenting, not homework
- Must be specific to the exact milestone (not generic "read more books")

One suggestion per milestone, also authored content. These are the single most valuable pieces of content in the entire parent experience — invest disproportionate effort in writing them well.

**Session history card**
Background: secondary surface
Label: "Recent activity results"
A list of the last 4 activity sessions for this milestone, each showing:
- Date and time
- Score (e.g. "3/3" or "2/3")
- Pass/fail indicator (green "✓ Pass" or amber "· Retry")

Display most recent first. Show maximum 4 entries — do not paginate in MVP.

This history serves two purposes: it shows the parent that the child is practising consistently (reassurance), and it shows that a single bad session doesn't derail progress (reducing anxiety when the child struggles one day).

**Footer note:**
Plain text below the last card: "X more passing sessions will achieve this milestone and unlock [next milestone name]."

This tells the parent exactly what to expect next, removing uncertainty.

## Screen 4: Milestone detail (achieved)

Reached by tapping any achieved milestone row on the area detail screen.

### Layout

Simpler than the in-progress detail. Contains:
- Header block: area + level label, milestone title, green "Achieved · [time ago]" badge
- "Why this matters" card (same as in-progress — the explanation is always shown, even after achievement)
- No session history — achievement is permanent, history is no longer relevant

No "try at home" for achieved milestones. The moment has passed. Instead, if there is a natural connection to the next in-progress milestone, a short contextual note can link the two: "Now that Aisha can recognise all her letters, she's starting to work on the sounds they make."

## Screen 5: Milestone achieved (celebration)

Reached by tapping a "Milestone reached!" feed item on the home screen.

This screen is distinct from the achieved milestone detail above — it is a celebratory entry point, reached in the moment of achievement, not as a retrospective lookup.

### Layout

**Celebration banner** at the top:
- Green background, green border
- Large star icon in a circle (36px, green)
- Bold title: "Aisha knows all 26 letters!" — specific and celebratory, not generic
- Subtitle: "Achieved yesterday · Language & Literacy"

**"What this means" card** (blue, same style as "Why this matters")
Explains the significance of this specific milestone in the context of the journey — what it unlocks, why it's a meaningful moment, not just a checkbox.

**"Celebrate it" card** (amber, same style as "Try at home")
A specific suggestion for marking the moment with the child. Not academic — about recognition, pride, and sharing. Examples:
- "Ask Aisha to show someone she loves that she knows all her letters."
- "Tell her teacher you heard about the achievement — she'll love knowing you know."
- "Tonight at dinner, ask her to count how many letters she can name. Watch her beam."

This card is unique to the celebration screen. It reinforces the emotional significance of milestones and positions the parent as someone who celebrates learning alongside their child.

## Screen 6: Activity result detail

Reached by tapping an activity completion feed item on the home screen.

### Purpose
Shows the parent exactly what their child did — not just a score, but the specific questions answered. This transforms an abstract "passed" into something the parent can picture and discuss.

### Layout

**Header:**
- Timestamp + location: "Today, 3:45 pm · completed at home"
- Activity name (plain language, 18px medium)
- Score badge + pass/fail badge side by side

**"Questions answered" card:**
A list of the 3 questions in the session, each showing:
- The question in simplified natural language ("Banana → B" rather than "matched word initial phoneme to grapheme")
- A tick or cross for correct/incorrect

This requires storing question-level results per session, not just pass/fail. The data model must record which specific variant was shown and what the child answered.

**"What she practised" card** (blue, same as "Why this matters")
One paragraph explaining what skill was practised and why it matters. This is the same "Why this matters" content from the milestone, reformatted slightly as a past-tense activity recap.

**Progress note:**
"X more passing sessions will achieve this milestone." Consistent with the milestone detail screen.

## Screen 7: Teacher messages

Reached by tapping the Messages tab, or by tapping a teacher note feed item on the home screen.

### Purpose
Async communication between parent and teacher. In the MVP, this is a simple message thread — no attachments, no read receipts, no group messaging.

### Layout

**Header:**
- Back arrow (if arrived from home feed) or no back (if arrived from tab)
- Teacher name: "Ms Priya"
- Role + class: "Class teacher, Nursery 2 Butterfly" — small, muted

**Message thread:**
Scrollable list of messages, chronological (oldest at top, newest at bottom).

Message bubbles:
- Teacher messages: secondary background, primary text, aligned left, bottom-left corner square (14px border-radius on all corners except bottom-left which is 4px)
- Parent messages: blue (#185FA5) background, white text, aligned right, bottom-right corner square
- Timestamp below each bubble, aligned to match the bubble

**Input row (fixed at bottom):**
- Text input: rounded pill shape, secondary background
- Send button: circle, blue, right arrow SVG icon

**Content guidance for teacher notes:**
Teacher notes that appear in the feed and thread should be warm and specific — the equivalent of what a good teacher would say at pickup. They are not assessment data. They are human moments. Examples of good notes:
- "Aisha was absolutely absorbed in the story corner today — she chose the same book three times."
- "Rayan helped a friend who was struggling with the puzzle this afternoon, unprompted."
- "We did a cooking activity today and she counted out the spoonfuls perfectly."

These notes exist separately from the observation logs that contribute to SED milestones. They are informal, relationship-building communication.

## Screen 8: P1 readiness detail

Reached by tapping the P1 readiness banner on the home screen.

### Purpose
Addresses Singapore parents' primary anxiety (P1 transition readiness) directly and honestly, with calibrating context that reduces rather than amplifies pressure.

### Layout

**Intro paragraph:**
This is the most important copy on this screen. It must be written and preserved exactly as intended:
> "This shows how Aisha is progressing toward the skills expected at the start of Primary 1. There's no rush — she has time. This is a guide, not a test."

The middle sentence ("There's no rush — she has time") must remain. It is the primary anxiety intervention. Do not shorten or remove it for space.

**Three area progress cards:**
One per learning area. Each contains:
- Area name (13px medium)
- Progress bar (8px, green fill) — this one CAN show proportional fill because it's a count of achieved/total milestones (a discrete, understandable number)
- Below the bar: "X of Y milestones achieved" (green, medium weight, left) + "Z in progress" (muted, right)

Note the deliberate asymmetry from the home screen area cards: here, showing the exact fraction (7 of 10) is appropriate because this screen is specifically for parents who want to understand P1 readiness. They have chosen to dig in — they can handle precision here. On the home screen, precision would be overwhelming.

**Calibrating note (bottom):**
A muted card with secondary background:
> "These milestones represent what children typically know and are able to do by the end of Kindergarten 2 — aligned to Singapore's NEL Framework. Aisha is in Nursery 2. She is on track."

The last sentence ("She is on track") is dynamic — it is only shown if the child's current progress is consistent with their age/year level. If a child is significantly behind, this sentence is replaced with: "Ms Priya can discuss Aisha's progress in more detail at your next check-in." This avoids false reassurance while still not alarming the parent through the UI — the teacher conversation is the right channel for concern.

## Screen 9: Activities tab

### Purpose
The child's personalised queue — for handing the device to the child to play. This screen belongs to the child, not the parent.

### Layout

**Header:** "Activities" (22px medium) + one-line description of what this is ("Aisha's personalised queue — updated weekly by Ms Priya.")

**Activity list:**
Cards for 2–3 assigned activities. Each card:
- If completed today: green background and border, checkmark icon, "Completed today" subtitle — visually distinct from incomplete
- If not yet completed: standard card, clock icon, "Ready to play" subtitle

Tapping an incomplete activity launches the child-facing activity experience (separate from the parent dashboard — this is a mode switch).

**Tone note below list:**
"Tap an activity to start it with Aisha, or let her tap it herself." — this surfaces both parent modes in one sentence: active parents co-participate, passive parents hand over the device.

## Feed logic and ordering

The home screen activity feed shows the 3 most recent events across all types, ordered by timestamp descending (most recent first). Types are interleaved — the feed does not group by type.

Events that appear in the feed:

| Event | Trigger |
|---|---|
| Activity completed | Child completes a session (any result — pass or retry) |
| Milestone achieved | ChildMilestoneProgress status changes to "achieved" |
| Teacher note | Teacher publishes a note to the parent feed |
| Behaviour observation (optional) | Teacher logs a SED observation (school configurable — may be kept internal) |

For activity completions, only show one feed item per milestone per day, even if the child completed multiple sessions. Use the most recent session's result. This prevents the feed from being dominated by a child who played the same activity 5 times in one afternoon.

Milestone achieved events should be shown even if they happened a few days ago — they are always worth surfacing. However, if a milestone achievement is older than 7 days, it is demoted below more recent events rather than shown in the top 3.

## Data requirements

The following data must be accessible to the parent dashboard. This is not a full data model — it is the read requirements for the dashboard specifically.

**Child profile:**
- First name
- Class name
- Teacher name
- Year level (N2 / K1 / K2)

**Per learning area:**
- Current level (Beginning / Developing / Secure)
- List of milestones with: ID, statement (plain language), status (not_started / in_progress / achieved), achieved_at timestamp
- For in-progress milestone: current passing session count, total sessions attempted, last session timestamp

**Per milestone (content, not user data):**
- Plain-language statement
- Parent-facing description
- "Why this matters" explanation text
- "Try at home" suggestion text
- Milestone type (skill / behaviour)

**Activity sessions (for feed and result detail):**
- Milestone ID
- Timestamp
- Location context ("at home" / "at school" — derived from whether the session was initiated from teacher or parent account)
- Pass/fail result
- Question-level detail: for each of the 3 questions, what was shown and what the child answered

**Teacher messages:**
- Full message thread between this parent and this teacher
- Each message: sender, body, timestamp

**P1 readiness aggregate:**
- Total milestones per area
- Achieved milestones per area
- In-progress milestone count per area

## Interaction states and edge cases

**Empty state — no activities completed yet:**
The home screen feed shows a friendly empty state: "Nothing logged yet this week. Aisha's activity queue is ready whenever she is." Do not show an empty list — an empty list reads as broken. The message should feel encouraging.

**New child — all milestones at Beginning, none achieved:**
The hero card reads: "Aisha is just getting started — her personalised learning journey begins this week." The area cards show Beginning badges with near-empty bars. This should feel like an exciting start, not a deficit.

**All Secure — child has achieved everything:**
A special hero card variant: "Aisha has achieved all her P1-readiness milestones. She's ready for Primary 1!" The area cards all show Secure with full green bars. The P1 banner says "All 30 milestones achieved." The activities tab shows extension activities (if implemented) or a congratulations message.

**Teacher hasn't sent any messages:**
The Messages tab shows a prompt: "Ms Priya will send updates through here. You can also message her if you have questions." Include teacher name dynamically.

**Child hasn't completed any activities in 5+ days:**
The hero card gently notes this: "It's been a few days since Aisha's last activity — a quick session this weekend could help keep her momentum going." Do not use language that implies failure or criticism of the parent. Keep it light.

**No internet connection:**
Show last-loaded data with a muted "Last updated [timestamp]" notice. Do not show error states for connectivity — parents should see something, not a broken screen.

## What this dashboard deliberately does not include

These omissions are as intentional as the inclusions. Do not add them without a deliberate product decision.

**Class averages or peer comparison.** See Rule 1. This is an absolute constraint.

**Percentage scores.** Activity results show fractions ("2 of 3") not percentages. Percentages invite comparison to arbitrary thresholds and feel like school tests.

**Red status indicators for the learning areas.** The Beginning badge uses coral/warm orange, not red. Red triggers alarm. Beginning is not alarming — it is the start of a journey.

**Notifications about missed sessions or streaks.** Streak mechanics and "you've missed 3 days" notifications create guilt and are counterproductive for parents who are already time-poor. Milestone progress speaks for itself.

**A graph of progress over time.** Historical charting is a Wave 2 feature (term trajectory reports). In the MVP, progress is current-state only. Showing a flat graph for a child who just started would be misleading; showing too short a window is meaningless.

**Teacher observation count for SED milestones** shown to parents as a number. The parent-facing SED milestone status shows "In progress" with a bar — not "3 of 5 observations." The count is internal to the teacher experience. Surfacing the count to parents would make them feel like their child is being scored on kindness, which is the wrong framing entirely.

## Content authoring requirements

The following content must be written by a human (curriculum specialist or teacher) and stored in the content database before the dashboard can function correctly. These are not generated dynamically in MVP.

For each of the 30 milestones:
1. Plain-language parent statement (already defined in milestone-schema.md)
2. "Why this matters" explanation (3–5 sentences, parent-facing, non-specialist language)
3. "Try at home" suggestion (1 specific activity, max 3 sentences)
4. Celebration text for the achievement moment (used on the milestone-achieved screen)

For each level-up transition (Beginning→Developing, Developing→Secure, per area):
5. A transition message explaining what the level change means and what comes next

For the hero card:
6. Template strings for each scenario (active week / quiet week / milestone week / new child / all achieved)

Total authored content items: approximately 30×4 + 6 + 6 = ~132 content items. This is a real scope item and should be planned as a content production workstream, not assumed to emerge from the tech build.
