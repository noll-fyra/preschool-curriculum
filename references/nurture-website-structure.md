# Nurture — Landing Page Website Structure
*Applies to all four persona landing pages: Teacher, Child, Parent, Schools*

---

## Overview

Each persona has its own dedicated landing page. All four pages share the same structural skeleton — the same section sequence, layout logic, and copy formula — but are populated with persona-specific copy, features, visuals, and emotional framing. This document defines the shared structure. The four persona copy files define the content that fills it.

---

## Page Structure (9 Sections)

### Section 1: Hero
**Purpose:** Immediate emotional resonance + clarity on what Nurture does for *this* persona. Must answer "is this for me?" within 3 seconds.

| Slot | Pattern |
|---|---|
| Product badge | Nurture wordmark + tagline |
| Headline | `[2–6 word emotional hook — speaks to the persona's core frustration or aspiration]` |
| Sub-headline | `[1–2 sentences: what Nurture does, for whom, and what changes]` |
| Primary CTA | `[Action verb] + [low-commitment framing]` e.g. "Get started free" / "Request a demo" |
| Hero visual | `[Product screenshot or illustration in context of this persona's experience]` |

**Tone rules per persona:**
- Teacher: Relief, time given back, professional respect
- Child: Joy, playfulness, curiosity — this page is for parents/teachers reading about the child experience
- Parent: Reassurance, closeness, informed
- Schools: Confidence, scale, outcomes

---

### Section 2: Category Positioning Line
**Purpose:** One line that names the product's space and the promise it makes to this persona.

> **Pattern:** `[Category], [resolution word — simplified / reimagined / done right / finally].`

This line sits between the hero and the first feature section. It acts as a bridge — validating that the visitor is in the right place and priming them for what comes next.

---

### Section 3: Primary Feature Showcase — 5 features
**Purpose:** Show Nurture doing the job this persona cares about. The five features should map to the five most important jobs-to-be-done for this persona.

**Layout:** 1 hero feature (full-width, large visual) + 4 supporting features in a 2×2 grid.

Each feature block:
- **Small icon** — area-specific, not decorative
- **Feature headline** — verb-led, outcome-oriented ("See every child's level at a glance")
- **1–2 sentence description** — what it does + why it matters to this persona
- **Product screenshot** — the feature in action, cropped to this persona's view

```
Feature 1 (hero, full-width):   [The single most important capability for this persona]
Feature 2 (grid):               [Supporting capability]
Feature 3 (grid):               [Supporting capability]
Feature 4 (grid):               [Supporting capability]
Feature 5 (grid):               [Supporting capability]
```

---

### Section 4: Integration / Connected Experience Section — 3 features
**Purpose:** Show how Nurture connects this persona to the rest of the learning loop. Reduce anxiety about adoption and switching.

> **Headline pattern:** `"[Verb phrase] with [what they already have / who they already work with]."`
> e.g. "Built to work with your classroom." / "Connected to your child's school, in real time."

Each of the 3 features shows a specific connection point:
- How Nurture connects to the persona's existing tools/context
- How data flows between personas (teacher → parent, parent → child, school → teachers)
- What the persona gains from being part of the loop

```
Feature 1 (hero, full-width):   [Primary integration / connection point]
Feature 2 (half):               [Secondary integration / connection point]
Feature 3 (half):               [Third integration / connection point]
```

---

### Section 5: Breadth / Range Section — 4 features
**Purpose:** Show that Nurture handles more than one context or use case for this persona. Broaden the value beyond the obvious primary job.

> **Headline pattern:** `"[Context A] and [Context B], [resolution]."`

Layout: 1 hero feature (full-width) + 3 in a row of thirds.

```
Feature 1 (hero, full-width):   [Cross-context or bridge capability]
Feature 2 (third):              [Use case variant 1]
Feature 3 (third):              [Use case variant 2]
Feature 4 (third):              [Use case variant 3 / flexibility / customisation]
```

---

### Section 6: Integrations / Ecosystem Grid
**Purpose:** Trust via ecosystem. Signal that Nurture is not an island — it fits into the world the persona already lives in.

> **Headline pattern:** `"Designed to fit into [their world / your workflow / your school / your day]."`

- 6 tiles in a 3×2 grid
- Each tile: `[Icon] + [Name] + [1-line description of what this connection enables]`
- For Schools/CTOs: reference curriculum frameworks, compliance, data formats
- For Teachers/Parents: reference familiar tools and workflows

---

### Section 7: Final CTA / Get Started
**Purpose:** Convert. Restate the value simply. Make the next step feel small.

> **Headline pattern:** `"[Start verb] [Product Name] [free / today / for your school]."`

- 1 sentence restating the core promise for this persona
- Two paths side-by-side: primary CTA (e.g. desktop app / web) + secondary CTA (e.g. mobile / request demo)
- Social proof line if available: "Trusted by X schools across Singapore"

---

### Section 8: Q&A
**Purpose:** Neutralize last-mile objections. This section is persona-specific — different personas have different concerns.

> **Headline:** `"Questions & answers"` — plain and direct

- 4–5 accordion questions
- Each question targets a specific objection type for this persona (see persona copy files for full Q&A sets)
- General objection categories covered across all personas:
  1. What exactly is Nurture / how does it work?
  2. How is it different from [what we have now]?
  3. What does it cost / is there a free version?
  4. Is it hard to set up / does it require training?
  5. Is our data safe? (especially relevant for Schools)

---

### Section 9: Footer
Standard footer grid. Sections: Product / For Schools / Resources / Company / Legal.
Social icons. App store badges (iOS + Android) for teacher and parent pages.

## Persona Navigation

Each page has a persistent nav bar with links to the other persona pages:
- "For Teachers" / "For Parents" / "For Children" / "For Schools"

This allows a headteacher who lands on the Teachers page to easily navigate to the Schools page, or a parent who arrives via a teacher referral to understand the full picture.

---

## Visual Identity Notes (shared across all pages)

- **Colour system:** Warm, optimistic palette. Avoid clinical blues and corporate greys. Nurture is an education product for young children — the visual language should feel safe, human, and forward-looking.
- **Typography:** Clear, friendly, readable at all sizes. Not playful to the point of looking like a toy. Not formal to the point of looking like compliance software.
- **Product visuals:** Each page shows the dashboard/view relevant to *that persona only*. A parent should never see the teacher dashboard; a teacher should never see the parent feed. Persona-specific screenshots are critical for trust.
- **The child page** is the exception: since children don't browse websites, the child-facing landing page is written *about* the child experience but *for* the parent or teacher reading it. The tone is warm and reassuring, the visuals show the child-facing app.
