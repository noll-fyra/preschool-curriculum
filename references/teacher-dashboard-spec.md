# Teacher Dashboard — Design Specification

## Purpose

This document describes the intended layout, behaviour, and day-long logic of the Teacher Dashboard for Nurture — a preschool education and learning platform. It is intended as a reference for AI coding assistants and developers building the frontend and backend of this feature.

---

## Core Design Principle

The dashboard is a **morning briefing, not a control panel.** A teacher opens it at ~7:45am with 10 minutes before children arrive. It must answer three questions instantly, without navigation:

1. **Who's here today?**
2. **What are we doing?**
3. **Who needs my attention?**

Everything else is secondary. The dashboard is **read-mostly** — it surfaces information. Actions are one tap away, not on the page itself.

The dashboard is also **time-aware**: it changes emphasis and surfaced content based on where the teacher is in the school day. There are five distinct phases (see Section 3).

---

## Section 1: Layout

### Structure
- **Persistent left sidebar** — navigation only; icons for Dashboard, Children, Schedule, Observations, Reports, Messages, Settings
- **Top bar** — class context and global actions
- **Two-column main area:**
  - **Left column (wide, ~65%)** — primary content; the most time-sensitive information
  - **Right column (narrow, ~35%)** — contextual content; supportive, quieter

### Top Bar Contents
- Left: Teacher's name + today's date
- Centre: **Class selector** — dropdown to switch between classes (relevant for teachers covering multiple groups); switching re-renders the full dashboard for that class
- Right:
  - **Quick-log button** — primary CTA; logs an observation; accessible from anywhere in the app
  - **Notifications bell** — with unread count badge
  - **Teacher avatar** — links to profile/settings

## Section 2: Dashboard Sections

### Left Column

#### 2.1 Stat Strip (top of left column)
Four stat cards displayed side by side. Each shows a single number with a label and one line of supporting context. No charts.

| Stat | Label | Supporting context |
|---|---|---|
| `16 / 19` | Children present | Names of absent children; whether parents have been notified |
| `5` | Activities today | Count done vs. remaining |
| `12` | Observations this week | Gentle nudge if below her usual cadence |
| `3` | Need attention | Flagged by system: observation gap, unread parent message, or domain not touched in 2+ weeks |

#### 2.2 Children Grid (centrepiece of the dashboard)
Every child in the class displayed as a card in a responsive grid. Visible all at once without scrolling.

Each child card shows:
- **Name and avatar** (initials with colour-coded background, consistent per child across the app)
- **Presence indicator** — dot in one of three states: `present` (green), `absent` (grey), `late` (amber) — updates in real time as children check in
- **Domain coverage dots** — a row of 5 small dots representing the five core developmental domains (Language & Literacy, Numeracy, Social-Emotional, Motor, Creative Arts). Each dot is filled if the child has had at least one observation in that domain in the current week; unfilled if not. This is not a progress score — it is an attention distribution indicator.
- **Flag icon** — appears if the system has flagged this child (see flagging logic below)

**Flagging logic:**
- No observation in any domain for 5+ days
- Unread parent message relating to this child
- A specific domain has had no observation for 10+ days
- Teacher has manually flagged the child

Tapping a child card opens the **Child Detail View** (a separate view, not a modal overlay).

#### 2.3 Today's Schedule (below children grid)
A vertical timeline of the day's activities — a sequenced list, not a calendar grid.

Each schedule block shows:
- Time and duration
- Activity name
- Learning domain tags (small coloured chips, e.g. "Language", "Motor")
- Status: `done` (visually subdued, struck through), `in progress` (highlighted in brand colour), `upcoming` (default)

The current block is always highlighted. Tapping any block opens the activity plan, materials checklist, and a quick-log prompt for observations against that activity.

---

### Right Column

#### 2.4 AI Insight (top of right column)
A single, one-paragraph nudge generated by the system. One insight at a time — not a list.

**Examples of insight types:**
- Observation gap: *"Amara hasn't had a motor skills observation in 12 days. Today's outdoor play block might be a natural moment."*
- Grouping opportunity: *"Three children are showing strong numeracy progress this week — consider a small group extension activity."*
- Domain imbalance: *"Creative arts observations are low this week across the class. Tomorrow's art station could address this."*
- Positive reinforcement: *"You logged 7 observations during this morning's outdoor block — your most productive session this week."*

Tapping the insight expands it with a suggested action and the supporting data behind it.

**Insight timing:** The content of this section changes based on phase of day (see Section 3). Morning = forward-looking. Afternoon = reflective.

#### 2.5 Quick Actions (2×2 grid)
The four most frequent actions that don't have a dedicated place in the main flow:

| Action | What it does |
|---|---|
| Log observation | Opens the quick-log sheet (same as top bar button) |
| Generate report | Opens report drafting tool, pre-filled with recent observations for a selected child |
| Send parent update | Opens compose view for individual or class-wide message |
| Plan tomorrow | Jumps to next day's schedule with AI activity suggestions |

#### 2.6 Developmental Snapshot (class-level)
Six horizontal bar charts — one per learning domain — showing how well-observed each domain is across the **whole class** this week (not per child). Expressed as a percentage of children observed in that domain, or a raw observation count.

Purpose: answers *"Am I overweighting literacy and neglecting creative arts?"* at a glance. No individual child data here.

#### 2.7 Recent Observations Feed
The last 5 observations logged — by this teacher or any co-teacher assigned to the class — in a compact list.

Each entry shows:
- Child avatar + name
- Domain tag (colour-coded)
- One-line excerpt of the observation note
- Timestamp
- Logged by (teacher name, if co-teaching)

This makes co-teaching visible in real time. A teacher can see what her assistant logged across the room without asking.

#### 2.8 Parent Messages (bottom of right column)
Unread parent messages only. Compact list showing:
- Parent name + child name
- Message preview (truncated)
- Timestamp

Tapping opens the full message thread. An unread badge also appears on the Messages icon in the left sidebar.

## Section 3: Time-Aware Behaviour — The Five Phases

The dashboard recognises the phase of day and adjusts what is emphasised and surfaced. Phase detection is based on the school's configured day schedule.

---

### Phase 1: Before School (default: 7:30 – 8:30am)
**Teacher mode: Preparation and awareness**

**Dashboard emphasis:**
- AI Insight is **forward-looking** — surfaces intention-setting nudges (*"Consider observing Kai in language during circle time today"*)
- Parent messages received overnight are **prominently surfaced** as a pre-class brief, not buried in the messaging section
- Schedule shows full day ahead; flags any activity with missing materials or preparation steps
- Stat strip shows expected attendance (based on any absence notifications received)

**Key behaviours:**
- Overnight parent messages appear in a dedicated "Before you start" callout at the top of the right column
- Materials readiness check: if an activity has a materials checklist, an incomplete checklist is flagged in the schedule
- Teacher can access and edit tomorrow's plan from the Quick Actions panel

---

### Phase 2: Morning Arrival (default: 8:30 – 9:00am)
**Teacher mode: Presence and check-in**

**Dashboard emphasis:**
- Children grid becomes the **primary focus** — presence indicators update in real time as children check in
- Each child's card may show a single context note pulled from recent parent messages or observation history (one line only — e.g. *"Parent flagged she was upset at drop-off yesterday"*)
- Absent children trigger an automatic status check: has the parent been contacted?

**Key behaviours:**
- Attendance can be marked directly from the children grid (tap to toggle presence state)
- The system auto-notifies parents of absent children if no prior notification has been received, after a configurable grace period
- Classroom display tools (noise meter, visual timer) are accessible from the top bar for use on a connected display screen

---

### Phase 3: Active Teaching (default: 9:00am – 12:30pm)
**Teacher mode: Observation and capture**

The teacher is with children. The dashboard steps back. The priority is **effortless capture with minimal interruption.**

**Dashboard emphasis:**
- Quick-log button is always visible and reachable in one tap from anywhere in the app
- Current schedule block is highlighted; tapping it shows the activity plan and a list of children the teacher intended to observe during that block
- After each schedule block ends, the system shows a **non-intrusive prompt**: *"[Activity name] just ended. Want to log anything?"*

**Quick-log flow (must complete in under 20 seconds):**
1. Tap quick-log
2. Select child (from grid or type name)
3. Speak or type a one-line observation note
4. System suggests domain tag — teacher confirms or overrides
5. Done — dismissed automatically

**Key behaviours:**
- Observations feed updates in real time — co-teacher observations appear as they are logged
- Behaviour/wellbeing incident logging is accessible from the child card; system auto-drafts a parent notification for teacher review
- The dashboard does not interrupt the teacher during active teaching; all prompts are dismissible in one tap

---

### Phase 4: Rest Time / Transition (default: 12:30 – 2:00pm)
**Teacher mode: Reflection and admin**

The teacher has a 20–40 minute window. The dashboard shifts to a **catch-up and review mode.**

**Dashboard emphasis:**
- A **"To review" list** surfaces at the top of the right column:
  - Observations logged this morning that are untagged or need domain confirmation
  - Unread parent messages
  - Report drafts ready for review (system has compiled recent observations into narrative form)
  - Children with zero observations today
- AI Insight is **retrospective** — reflects on the morning: *"You logged 7 observations during outdoor play — your highest this week."*
- Developmental snapshot is most actionable here — shows today's domain coverage and highlights gaps

**Key behaviours:**
- Bulk observation tagging: teacher can review and confirm AI-suggested domain tags for multiple observations in a single scrollable flow
- Report review: system-drafted narrative reports are presented for each child; teacher edits and approves, does not write from scratch
- Zero-observation children list is presented as a prompt, not a warning — teacher acknowledges or schedules observation for tomorrow

---

### Phase 5: End of Day / After School (default: 3:00pm onwards)
**Teacher mode: Closure and handover**

Three jobs need to happen. The dashboard orchestrates all three in sequence.

**Job 1: Daily parent updates**
- System has compiled a draft daily update for each child — drawn from today's observations, activity participation, and any logged photos
- Teacher reviews each draft, makes quick edits, and approves for sending
- These are not written from scratch — they are 19 drafts that require ~15 seconds each to approve or adjust
- Bulk approve is available for days when the teacher is short on time

**Job 2: Handover notes**
- A free-text handover field for notes to co-teachers or for the next school day
- Example: *"Lila had a hard afternoon — watch her arrival tomorrow"*
- Handover notes surface in the Phase 1 (Before School) view the following morning

**Job 3: Tomorrow's plan**
- Next day's schedule is shown with any gaps or unplanned blocks highlighted
- AI suggests activities to fill gaps, based on recent developmental data: *"Consider a fine motor activity — four children haven't had one this week"*
- Teacher accepts, modifies, or dismisses suggestions

**End-of-day summary (bottom of dashboard):**
A simple, non-judgmental closure view showing:
- Total observations logged today
- Children observed vs. not observed
- Open parent messages or follow-up actions
- Confirmation that daily updates have been sent

## Section 4: What Is Deliberately Excluded

The following are **not on the dashboard** and should not be surfaced here:

- Billing, fee management, or payment status
- Full attendance registers (the grid covers presence; full register is in Admin)
- Staff schedules or HR information
- Lesson plan editing (accessible from the schedule, not the dashboard)
- Individual child deep-dives (accessible by tapping a child card)
- School-level analytics or programme-wide reporting
- Anything that requires scrolling to find on first load

The dashboard is a daily operational view. Administrative, financial, and programme-level tools live in separate sections of the app.

---

## Section 5: Underlying Logic Summary

Every section on the dashboard answers a specific teacher question:

| Section | Question answered |
|---|---|
| Stat strip | Is today going normally? |
| Children grid | Have I been paying attention to the right people? |
| Today's schedule | What am I doing and where am I in the day? |
| AI insight | What am I missing that I wouldn't notice myself? |
| Developmental snapshot | Am I covering the curriculum broadly enough? |
| Observations feed | What has my team already captured today? |
| Parent messages | Is there anything I need to respond to before class? |

---

## Section 6: Key Design Constraints for Implementation

- **Performance:** The children grid and observations feed must load instantly. Both are the first things a teacher's eye goes to. Lazy loading or skeleton states are acceptable; blank states are not.
- **Offline capability:** Quick-log must work offline and sync when connection is restored. Teachers move around classrooms that may have poor connectivity.
- **One-tap access to quick-log:** This is the single most frequent action in the app. It must never require more than one tap to open from any screen.
- **Phase detection:** Phase transitions are based on the school's configured daily schedule, stored per class. They should not trigger hard UI resets — changes should feel like gentle re-emphasis, not jarring rebuilds.
- **Co-teaching awareness:** All real-time data (observations feed, attendance, messages) reflects activity from all staff assigned to the class, not just the logged-in teacher.
- **No friction on dismissal:** Every AI insight, prompt, and nudge must be dismissible in one tap. The system offers; the teacher decides.
- **Child data privacy:** No child's full name, photo, or developmental data should appear in notifications, browser tab titles, or any surface visible outside the authenticated app session.
